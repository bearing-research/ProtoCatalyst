# Truffle as an AOT-clean execution backend ‚Äî an exploration plan

*Status: **planning / doc-only**. No code yet. This is the executable counterpart to
[`../scala3-encoder/AOT_ROADMAP.md`](../scala3-encoder/AOT_ROADMAP.md) ¬ß6 **Option B** ("Re-express
Catalyst as a GraalVM Truffle language"). The roadmap argues Truffle is a plausible-but-unproven way
to make Spark's query execution AOT-safe without runtime bytecode generation, and flags the central
unknown as **"a risk, not a measurement."** This document is the plan to **turn that risk into a
measurement** ‚Äî cheaply, and in a way that keeps a path to Spark.*

---

## ¬ß0. Why do this at all (and why here)

Spark executes SQL by generating Java source at query time and compiling it to bytecode with **Janino**
(whole-stage codegen, WSCG). That runtime bytecode generation is the load-bearing blocker for
GraalVM `native-image` (closed-world: no runtime codegen) ‚Äî blocker #2 in the AOT inventory, the one
with **no documented workaround**. The encoder work (blocker #1) unblocks the *typed Dataset* and
*Connect client* paths; it does nothing for the executor, whose obstacle is codegen.

[Truffle](https://www.graalvm.org/latest/graalvm-as-a-platform/language-implementation-framework/) is
Oracle's framework for self-specializing AST interpreters that Graal's **partial evaluation** (PE)
JIT-compiles at runtime. Crucially, Truffle interpreters are **designed to run under native-image** ‚Äî
it is how GraalVM ships JS, Python, Ruby, R, and LLVM as AOT binaries. So a query engine expressed as
Truffle nodes would:

- build a query plan **at runtime** (allocation + tree-rewriting ‚Äî legal under native-image),
- compose it from **pre-existing** `Node` types (no freshly generated bytecode),
- and let Graal PE specialize it ‚Äî **all AOT-legal**.

That is the one option in the roadmap that preserves **dynamic SQL** (the `spark.sql(userInput)` path
that compile-time codegen, Option A, structurally cannot cover) *and* is AOT-clean.

**Why ProtoCatalyst is the right vehicle ‚Äî Truffle is an isolated new backend, not a rewrite.** We
already own every piece Truffle needs to plug into:

| Asset | Role for this exploration |
|---|---|
| `ProtoPhysicalPlan` / `ProtoExpr` IR | the plan to compile *from* (the Truffle AST builder's input) |
| `PhysicalPlanExecutor` + `ExprEvaluator` (columnar Scala interpreter over Arrow) | the **perf baseline to beat** and the reference semantics to copy |
| DataFusion backend (native, vectorized, runs the same plans) | the "what good looks like" reference point |
| `TpchCrossBackendSuite` (12 cases, row-parity) | correctness **almost for free** ‚Äî Truffle becomes a 3rd backend behind `execute(plan)` |

The Truffle backend sits behind the exact same `execute(plan)` seam as Local Arrow and DataFusion
(see [`CROSS_BACKEND.md`](CROSS_BACKEND.md)). It touches **none** of the encoder / cross-version build.

**The final goal is Spark.** This prototype runs on *our* IR, but every design choice below is made so
the result maps onto Catalyst ‚Äî parity and the migration path are first-class (¬ß6, ¬ß7), not
afterthoughts. The prototype's job is to produce the evidence (perf number + AOT existence proof) that
would justify, or kill, a real Spark effort.

## ¬ß1. The central question to design around

Truffle's published headline is "**~1.5‚Äì3√ó of hand-tuned compiled code**" on language-interpreter
workloads. That figure is a *fraction of a ceiling*, and which way it cuts depends entirely on the
baseline:

- **Python/Ruby:** the realistic alternative is a naive tree-walker (10‚Äì100√ó slower). Truffle climbs
  most of the way to the ceiling ‚Äî a huge **win**.
- **Spark:** WSCG already *is* that ceiling (fused Java ‚Üí near-hand-tuned machine code). The same
  "1.5‚Äì3√ó of hand-tuned" reads as a **1.5‚Äì3√ó regression** ‚Äî subtracting from a fast baseline, not
  climbing toward it.

Two compounding reasons it would likely land *below* WSCG specifically:

1. **Specialization may not pay off.** Truffle's core mechanism is runtime self-specialization on
   *observed* types. But a query engine already knows every column's type from the **schema at plan
   time** ‚Äî so that machinery mostly adds overhead instead of buying anything.
2. **Fusion gap.** WSCG fuses operators into one loop with no intermediate row materialization.
   Truffle walks an AST of nodes; PE flattens much of that but is **not guaranteed** to recover the
   same fused-loop structure and register allocation.

**The unexplored escape hatch ‚Äî and the actual research contribution ‚Äî is node granularity.** The
canonical Truffle shape is **row-at-a-time** (one `executeRow` per node per row), which is exactly the
shape the literature predicts will regress. But nobody has run Truffle on a **vectorized** engine. A
**batch-at-a-time** node processes a whole Arrow column/batch per `execute` call, amortizing node
dispatch over thousands of rows and exposing SIMD-friendly inner loops ‚Äî the DuckDB/Photon/Velox move,
the one that closes the gap to WSCG. **Whether batch-shaped Truffle nodes can be competitive over an
already-high baseline is the open question this exploration answers.** We build *both* shapes; the
head-to-head between them is the result.

> **Honest framing.** We adopt Truffle for **AOT-safety + dynamic-SQL support**, and the price is
> (probable) throughput. The exploration's deliverable is the *magnitude* of that price, measured ‚Äî
> including the possibility that the answer is "too much," which is itself the first published data
> point for Truffle on a big-data engine.

## ¬ß2. Language decision ‚Äî Java node library + Scala builder (hybrid)

**Decision: the Truffle `Node` classes must be Java; the AST builder and harness integration stay
Scala.** This was researched, not assumed.

The Truffle DSL is a **Java annotation processor**. You write abstract `Node` classes with
`@Specialization`-annotated methods (and `@NodeChild`, `@TypeSystem`, guards); the processor generates
the concrete `‚Ä¶NodeGen` subclasses that implement the *uninitialized ‚Üí specialized ‚Üí generic* state
machine. That generated code is the entire point of the DSL ‚Äî it removes the hand-written
state-machine boilerplate.

The blocker: **annotation processors are not supported in Scala** (the Scala compiler does not run
Java's `javax.annotation.processing` pipeline). So a Scala module *cannot* host the Truffle DSL. The
only ways to use Truffle from Scala are (a) hand-write every specialization state machine yourself ‚Äî
re-introducing exactly the boilerplate the DSL exists to delete, and going off the supported path ‚Äî or
(b) don't; write the nodes in Java.

There is research on *"Specializing Scala with Truffle"* (Waterloo, UWSpace) but that is about using
Truffle to JIT *Scala programs*, **not** about authoring Truffle nodes in Scala ‚Äî it does not change
this conclusion.

**Resolution that honors "prefer one language":**

- **`truffle-exec` (new module, Java).** The `Node` library: expression nodes, operator nodes, the
  `TypeSystem`, frame/columnar value classes. Java is *required* here ‚Äî but only here.
- **The `ProtoPhysicalPlan ‚Üí Truffle AST` builder, the `execute(plan)` backend adapter, and all tests
  stay in Scala.** Scala calls into the generated Java `‚Ä¶NodeGen` factories with zero friction
  (ordinary Java interop). The builder is plain tree-walking ‚Äî no annotations, no DSL.

So the language split is *minimal and load-bearing*: Java exactly where the framework demands it, Scala
everywhere else. This also mirrors the existing `proto` (Java) + Scala-consumers pattern in the repo.

- **`TruffleBackend.execute(plan)`** (Scala): walk `ProtoPhysicalPlan` + `ProtoExpr`, instantiate the
  Java `‚Ä¶NodeGen` nodes, wrap in a Truffle root node, get a `CallTarget`, run it over the input Arrow
  `VectorSchemaRoot`s, collect the output `Batch`.
- **Two node shapes behind one builder** (the experiment): a `row` mode and a `batch` mode, selectable
  so the same plan runs both ways for an apples-to-apples comparison.
- **Data representation.** Input/output are Arrow vectors (same as the Local backend), so parity
  comparison reuses the harness's `readRows` / `rowsAgree` unchanged. Batch nodes read Arrow buffers
  directly; row nodes iterate.

## ¬ß4. The phased plan (kill-experiment first)

Ordered so the **cheapest experiment that can kill the idea runs first**, and native-image (the
expensive payoff) runs only if the perf number survives.

### Phase 0 ‚Äî Toolchain & skeleton
- Add **GraalVM** (with the **optimizing Truffle runtime** ‚Äî *not* the fallback interpreter runtime,
  which doesn't partial-evaluate and would yield meaningless numbers) for the `truffle-exec` module
  only. The rest of the build stays on JDK 21 via `.sbtopts`; this module is isolated.
- Wire the Truffle DSL annotation processor into the Java module's sbt config.
- Prove a "hello node" **PE-compiles** (confirm Graal is actually optimizing it, e.g. via
  `-Dpolyglot.engine.TraceCompilation=true`). De-risk the boring infra before writing real nodes.

> **Status: DONE.** Module `truffle-exec` (Java, out of the root aggregate) with `truffle-api` +
> `truffle-dsl-processor` (25.0.2). Findings: (1) the DSL annotation processor runs inside our sbt
> build ‚Äî `AddNode @Specialization` ‚Üí generated `AddNodeGen`, no special wiring needed. (2) **The
> optimizing runtime requires a GraalVM JDK.** Stock homebrew openjdk@21 reports the `Interpreted`
> runtime even with `truffle-runtime` + `truffle-compiler` on the classpath and `-XX:+EnableJVMCI`
> (homebrew's OpenJDK lacks the JVMCI-compiler integration). On Oracle GraalVM 21 the runtime is
> `Oracle GraalVM` and the hello-node compiles through Tier 1 + Tier 2 (PE confirmed). The module
> pins `javaHome` to GraalVM 21; the rest of the build stays on openjdk@21.

### Phase 1 ‚Äî Minimal vertical slice
- Expression nodes for ~6 ops: `Literal`, `ColumnRef`, `Add`, `Multiply`, `LessThan`, `And`.
- Operator nodes: `Project`, `Filter`.
- The `ProtoPhysicalPlan ‚Üí AST` builder (Scala), in **both** row and batch shapes.
- Target **Q6's filter+project** ‚Äî already exercised by the harness, so the oracle exists.

### Phase 2 ‚Äî Measure under JIT (the go/no-go gate)
- Benchmark on GraalVM **JIT first** (no native-image yet): `Truffle-row` vs `Truffle-batch` vs the
  Scala columnar interpreter vs (stretch) Spark WSCG on equivalent data, JMH, same methodology as
  [`../scala3-encoder/BENCHMARKS.md`](../scala3-encoder/BENCHMARKS.md).
- **Gate:** if `Truffle-batch` cannot get within **~1.5√ó** of the Scala interpreter under JIT, the
  regression story dominates ‚Üí **stop and write up the negative result** (still the first such data
  point for a vectorized engine). If it clears the bar ‚Üí continue.

> **Status (preliminary, Q6 filter‚Üíproject slice).** GraalVM 21, JMH `-f2 -wi5 -i8`, avg ¬µs/op (16
> obs). `rawJava` = a hand-written fused loop (WSCG proxy / ceiling). Three Truffle shapes: `row`
> (fused row-at-a-time), `sel` (selection-vector vectorized ‚Äî the Photon/Velox model), `batch` (naive
> full-materialization vectorized).
>
> | rows | rawJava | row (fused) | sel (selection-vec) | batch (naive) |
> |---|---|---|---|---|
> | 4096 | 3.69 | **6.05 (1.64√ó)** | 8.02 (2.18√ó) | 15.59 (4.2√ó) |
> | 65536 | 59.8 | **95.8 (1.60√ó)** | 133.6 (2.23√ó) | 237.3 (4.0√ó) |
>
> Findings: (1) **Fused row-at-a-time Truffle is ~1.6√ó the raw ceiling** ‚Äî squarely in Truffle's
> published band, and an encouraging AOT result (an AOT-safe interpreter within ~1.6√ó of codegen).
> (2) The hypothesis that *batch* would be the contribution was **inverted, then partly recovered**.
> Naive batch is ~4√ó the ceiling ‚Äî not allocation (node-owned reusable buffers fixed that; variance
> collapsed) but **memory traffic**: ~7 full-column passes vs the fused loop's single
> register-resident pass. (3) **Selection vectors ‚Äî the fair test of vectorization ‚Äî recovered most of
> that loss** (4√ó‚Üí2.2√ó, and from 2.4√ó slower than row to ~1.35√ó), but **fusion still wins for this
> query**: for a cheap, memory-bound 3-predicate-and-a-multiply, the fused single pass beats
> selection-pushdown's multiple passes + gather/scatter + bookkeeping. Selection vectors pull ahead
> when predicates are very selective early, per-tuple compute is heavier, or pipelines are long ‚Äî
> none true for Q6. **Defensible conclusion: vectorization got its best shot and fusion still wins for
> this query shape; the AOT headline is fused Truffle at ~1.6√ó codegen.** Still outstanding for the
> gate: the Scala-interpreter baseline (needs the AST builder).
>
> **Crossover experiment (does vectorization ever overtake fusion?).** A heavier branchless projection
> ({@code r = ep*d; r*=2-d; r*=1+d*d}) across two selectivities (`Q6HeavyBench`, 65536 rows, ¬µs/op):
>
> | dataset | rawJava | row (fused) | sel (sel-vec) | batch (naive) |
> |---|---|---|---|---|
> | selective (~13% pass) | 61.9 | **98.6 (1.59√ó)** | 129.9 (2.10√ó) | 266.6 (4.3√ó) |
> | allpass (~100% pass) | 52.2 | **142.5 (2.73√ó)** | 284.5 (5.45√ó) | 313.0 (6.0√ó) |
>
> The crossover **does not happen** ‚Äî the opposite of the SIMD hypothesis: in the projection-dominated
> `allpass` case `sel` gets *worse* relative to `row` (2.0√ó vs 1.3√ó). Root cause: our vectorized shapes
> do **one pass per node**, and the filter alone is ~4 passes (scan + 2 refines + gather) vs the fused
> loop's single register-resident pass. **To beat fused row a vectorized engine needs kernel fusion ‚Äî
> compiling several ops into one SIMD loop ‚Äî i.e. exactly the runtime codegen AOT forbids.** So for an
> AOT-safe, no-codegen interpreter, **fused row-at-a-time is the winning shape**, and it sits ~1.6‚Äì2.7√ó
> the codegen ceiling depending on how much of the work clears the filter. (Caveat: a *hand-fused*
> vectorized kernel ‚Äî one node doing filter+project+SIMD in a single pass ‚Äî was not built; that is the
> Velox/Photon design and it reintroduces per-shape kernel generation.)

#### Path to Catalyst parity (capability layers)

Where the Phase-3 milestone sits (numeric-double Filter/Project) ‚Üí full Catalyst, ordered by leverage.
Each layer is independently testable against the project's interpreter and, ultimately, **Spark as the
correctness oracle** (the same discipline the encoder work uses, via `spark-catalyst` + golden fixtures).

1. **Typed value model + nulls ‚Äî the foundational rewrite.** Replace the double-only model with typed
   reads (long / double / decimal / boolean / string / date / timestamp) directly over Arrow vectors,
   plus three-valued null logic (Arrow validity bitmaps ‚Üí null propagation). **This is where the
   Truffle DSL earns its keep:** `@Specialization` per operand type (e.g. `Add` specialized for
   long/double/Decimal) is exactly the uninitialized‚Üíspecialized‚Üígeneric machinery the DSL generates
   (Phase 0 set it up). Everything downstream depends on this; it is the make-or-break layer.
   **Started:** the typed/null-aware DSL core is built and validated ‚Äî `truffle-exec/typed`
   (`SqlTypes` `@TypeSystem` with `long‚Üídouble` implicit cast, `TExpr` base over `VirtualFrame`,
   nullable typed columns, `@Specialization` arithmetic/comparison, hand-written 3VL `AND`/`OR`).
   `TypedNodesSpec` proves typed long/double dispatch, `long‚Üídouble` promotion, null propagation, and
   `TRUE AND NULL` / `FALSE OR NULL` correctly failing WHERE. Null is threaded through the DSL via
   `UnexpectedResultException` on the typed path ‚Üí fallback to the generic/NULL specialization.
   **Now wired to real plans:** `TypedTruffleCompiler` compiles a `Filter` over a `TableScan` into the
   typed nodes, decoding Arrow columns *with their validity bitmaps* (`TypedColumnsDecoder`).
   `TypedBackendParitySpec` shows it matching the interpreter's 3VL on **nullable** data ‚Äî where the
   double-only `GNodes` path can't even run (Arrow's `get()` throws on a NULL slot). Typed
   **multi-column projection output** is also done ‚Äî `TypedFilterProjectRoot` writes null-preserving
   boxed `Long`/`Double`/`null`, validated by a projection that keeps NULL cells and agrees with the
   interpreter. And **null-aware global aggregate** (`TypedGlobalAggRoot`): SUM/AVG/MIN/MAX skip NULL
   inputs and yield NULL when none seen, COUNT(*) counts surviving rows ‚Äî validated against the
   interpreter, including `SUM` ignoring NULLs and `SUM` over empty input being NULL (not 0). What's
   left for Layer 1 is breadth of **column types**: string, decimal (exact), date/timestamp ‚Äî each a
   new `T*Column` kind + `SqlTypes` member + decode case. The numeric/null core is complete.
2. **Expression coverage (~10 ‚Üí ~93).** Cast (real coercion rules), CASE/IF/Coalesce/NullIf, In, Like,
   string/date/math functions, decimal arithmetic with precision/scale. Mechanical but large; each
   needs Catalyst-exact semantics, not just a plausible implementation.
3. **Stateful operators.** Global + grouped aggregate (HashAggregate accumulator model), Sort, Limit,
   Distinct, the three joins, Window. Aggregates and joins are the structural lifts (state, later
   spilling).
4. **Semantics parity (the oracle).** Match Spark exactly: type coercion, decimal precision/scale,
   null propagation, overflow (ANSI vs legacy), collation, rounding. Validated against Spark.
5. **Front-end swap ‚Äî the actual Spark graft.** Replace the `ProtoPhysicalPlan` builder with one
   consuming Catalyst's `SparkPlan`/`Expression` trees directly; **reuse the entire node library
   unchanged**. This is the step that turns the prototype into a Spark executor (¬ß6 migration map).
6. **Scale-out + native-image.** Wire every TPC-H query through the cross-backend harness;
   native-image the full backend; benchmark cold-start + throughput at scale.

**Sequencing.** Layers 1‚Üí4 are the engine (the "2‚Äì3 year" estimate in
[`../scala3-encoder/AOT_ROADMAP.md`](../scala3-encoder/AOT_ROADMAP.md) ¬ß6 Option B); Layer 5 is the
Spark graft; Layer 6 is hardening. Near-term leverage order: a fused **global aggregate** (next
increment) unlocks real Q6; **decimals + grouped aggregate** unlock Q1; then **Layer 1 (typed values +
nulls) is the gate to everything else** and should land before breadth (Layer 2) to avoid rework. The
honest bottleneck is not operator breadth ‚Äî it's that full parity is fundamentally a *typed,
null-aware, Spark-semantics-exact* system, and Layers 1 + 4 are the bulk of the multi-year cost.

### Phase 4 ‚Äî The AOT payoff (the reason this is worth it)
- `native-image` a `parse ‚Üí optimize ‚Üí Truffle AST ‚Üí execute` driver and prove **dynamic SQL runs in a
  native binary with zero runtime codegen** ‚Äî the existence proof ¬ß6/Option B says does not exist.
- Measure **cold start** (native vs JVM) and steady-state throughput.

> **Status: DONE ‚Äî and the result exceeds the goal.** `Q6Native` builds the fused-row Q6 plan *at
> runtime* (a Truffle AST composed at runtime ‚Äî the analogue of planning `spark.sql(userInput)`) and
> executes it. Built with:
> ```
> GRAALVM=/Library/Java/JavaVirtualMachines/graalvm-21.jdk/Contents/Home
> CP=$(sbt -batch "export truffleExec/runtime:fullClasspath" | tail -1)
> "$GRAALVM/bin/native-image" -cp "$CP" protocatalyst.truffle.Q6Native -o q6native --no-fallback
> ```
> Outcome (39 MB Mach-O arm64, built in ~74 s):
> - **A runtime-built query plan runs in a closed-world native binary with zero runtime bytecode
>   generation** ‚Äî the Janino/WSCG blocker (#2) sidestepped, on the *dynamic* path Option A can't cover.
> - **The optimizing runtime survives into the image.** The binary reports `runtime : Oracle GraalVM`
>   (not "Interpreted") ‚Äî the Graal compiler + Truffle runtime are embedded (`jdk.internal.vm.compiler`,
>   `org.graalvm.truffle.runtime.svm` in the image), so **partial evaluation works in-binary**, not
>   just on the JVM. This is the crux: AOT-safe *and* JIT-quality execution at once.
> - **Cold start ~30√ó faster than the JVM.** Whole-process wall time: native **~10 ms** (warm FS;
>   0.61 s on first cold-page-cache run) vs the same code on `java` at **~310‚Äì510 ms**. Correct result
>   (8582 matches at 65536 rows).
>
> This is the published-case-study-grade result: the first demonstration (to our knowledge) of a
> runtime-planned, vectorizable query engine executing inside a GraalVM native image with runtime
> compilation intact.

## ¬ß5. Parity strategy (correctness)

Parity is non-negotiable for a Spark-bound result. Two layers:

1. **Cross-engine parity (free, immediate):** the Truffle backend must agree row-for-row with the
   Local interpreter and DataFusion on all harness cases. This is the same yardstick the project
   already trusts for the engine-independence thesis.
2. **Catalyst-semantics parity (the Spark-facing one):** the node implementations must match Spark's
   expression semantics where they differ from naive arithmetic ‚Äî three-valued null logic, type
   coercion / implicit casts, **decimal precision/scale** (the Q6 decimal-overflow divergence already
   documented in the harness is a live example), date/time, string collation. The reference is
   `ExprEvaluator` (which already encodes our parity choices) plus Spark's `Expression` semantics. Each
   node carries a doc note where it deliberately matches a Catalyst quirk.

## ¬ß6. The Spark migration path (the point of the prototype)

The prototype runs on `ProtoPhysicalPlan`. A real Spark integration **reuses the same Java node
library** and swaps only the *front-end builder*:

```
Prototype:   ProtoPhysicalPlan / ProtoExpr  ‚îÄ‚îÄbuilder‚îÄ‚îÄ‚ñ∫  Truffle AST  ‚îÄ‚îÄPE‚îÄ‚îÄ‚ñ∫  native-clean exec
Spark:       SparkPlan / Catalyst Expression ‚îÄ‚îÄbuilder‚îÄ‚îÄ‚ñ∫  Truffle AST  ‚îÄ‚îÄPE‚îÄ‚îÄ‚ñ∫  native-clean exec
                         (swap this)            (reuse)        (reuse)
```

- **Vocabulary mapping.** Catalyst's expression layer is ~93 `Expression` types and ~25 plan
  operators (the same inventory Option A faces). The prototype covers the TPC-H-relevant subset; the
  doc tracks a mapping table (our `ProtoExpr` case ‚Üî Catalyst `Expression` ‚Üî Truffle node) so the gap
  to full Catalyst coverage is always explicit.
- **Where Truffle plugs into Spark.** It replaces `WholeStageCodegenExec`'s *output* ‚Äî i.e. it is an
  alternative to the codegen path, selected for the native-image build profile, falling back to (or
  coexisting with) interpreted/WSCG modes elsewhere. It does **not** require touching the parser,
  analyzer, or optimizer ‚Äî those build the `SparkPlan` that the Truffle builder consumes.
- **The migration prize.** Once the node library exists, a Catalyst front-end builder yields an
  AOT-clean Spark executor that runs **dynamic SQL** in a native binary ‚Äî the capability Option A
  cannot provide and Leyden does not target. That is the unique thing Truffle buys, and the reason it
  is worth measuring despite the throughput risk.

## ¬ß7. Risks & open questions (carry these honestly)

- **Throughput regression over WSCG** ‚Äî the ¬ß1 question. *Mitigation:* batch nodes; measure before
  investing; kill-gate at Phase 2.
- **Specialization adds overhead, not value** ‚Äî schema types are known at plan time, so Truffle's
  self-specialization may be dead weight. *Open:* whether batch-shaped nodes can sidestep this by
  specializing per-*column-type* once rather than per-row.
- **Fusion gap** ‚Äî PE may not recover WSCG's single-loop structure. *Open:* does batch granularity make
  this moot (the loop is inside the kernel, not across nodes)?
- **Strategic coupling.** Truffle is Oracle-controlled (part of GraalVM). A Spark dependency on it
  couples a core layer to one vendor's framework ‚Äî the ASF historically resists this. This is a
  *project/governance* risk independent of whether it works technically; flag it in any upstream pitch.
- **Engineering cost to full Catalyst coverage** ‚Äî comparable to a Catalyst rewrite (93 expr types √ó
  specialization slots + 25 operators). The prototype deliberately does **not** attempt this; it proves
  the *mechanism* on a slice and measures it.

---

## ¬ß8. What this exploration is and is not

- **Is:** a scoped experiment to measure Truffle's perf over our high baseline, prove AOT-clean dynamic
  SQL in a native binary, and keep a concrete map to Spark. Doc-only until approved to build.
- **Is not:** a commitment to a full Catalyst-on-Truffle rewrite, nor a claim that Truffle beats WSCG.
  The roadmap rates Option B "research-grade, 2‚Äì3 year effort if a Truffle-experienced group took it
  on." This prototype is the small front-loaded experiment that tells us whether that larger effort is
  worth proposing.
