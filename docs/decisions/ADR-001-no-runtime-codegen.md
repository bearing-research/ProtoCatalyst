# ADR-001: No Runtime Code Generation

## Status

**Accepted** - January 2026

## Context

Spark's `ExpressionEncoder` uses Janino to generate specialized Java bytecode at runtime for serialization/deserialization. This raises the question: should ProtoCatalyst implement similar runtime code generation?

### What Spark's Codegen Does

Spark generates specialized code like:
```java
// Generic (slow - boxing, virtual dispatch)
for (int i = 0; i < schema.size; i++) {
  Object value = row.get(i);
  write(value, schema[i].dataType);
}

// Generated (fast - direct field access, no boxing)
void serializePerson(Person p, UnsafeRowWriter writer) {
  writer.writeUTF8String(0, UTF8String.fromString(p.name()));
  writer.writeInt(1, p.age());
}
```

### Current ProtoCatalyst Approach

ProtoCatalyst uses Scala 3's compile-time metaprogramming:
- `inline` methods for compile-time expansion
- `Mirror.ProductOf` for type-safe field access
- `productElement(i)` for runtime field access

## Decision

**We will NOT implement runtime bytecode generation (Janino/ASM).**

Instead, we will:
1. Continue using Scala 3's `inline` for compile-time specialization
2. Rely on JIT optimization for hot paths
3. Consider inline type specialization as a future enhancement if needed

## Rationale

### 1. Already Faster Than Spark

Benchmark results (from ENCODER_DEEP_DIVE.md):

| Operation | Proto (μs) | Spark (μs) | Speedup |
|-----------|-----------|-----------|---------|
| serializePerson | 0.028 | 0.073 | **2.6x** |
| serializeComplex | 0.035 | 0.253 | **7.2x** |

ProtoCatalyst is faster WITHOUT codegen because:
- Schema derived at compile time (not runtime)
- No expression tree interpretation
- Simpler data structures

### 2. Scala 3 Provides Better Alternative

| Aspect | Spark Codegen | Scala 3 Inline |
|--------|--------------|----------------|
| When | Runtime | Compile time |
| Startup | JIT warmup needed | Zero overhead |
| Debugging | Generated code | Source visible |
| Dependencies | Janino | None |

### 3. Complexity vs Benefit

Adding Janino would:
- Increase dependencies
- Complicate debugging
- Add maintenance burden
- Provide marginal benefit given existing performance

### 4. JIT Already Optimizes

The `productElement(i)` hot path is optimized by the JVM JIT compiler after warmup. Modern JVMs inline small methods and eliminate virtual dispatch for monomorphic call sites.

## Consequences

### Positive

- Simpler codebase
- No external code generation dependencies
- Easier debugging (no generated bytecode)
- Compile-time type safety preserved
- Already faster than Spark

### Negative

- `productElement(i)` still has boxing overhead for primitives
- Cannot generate truly specialized code per case class
- May leave ~10-20% performance on the table for extremely hot paths

### Future Options

If performance becomes critical, we can:

1. **Inline Type Specialization** (Medium effort)
   - Extend `inline` derivation to specialize on field types
   - Eliminate runtime type matching in `serializeField`

2. **Macro Direct Field Access** (High effort)
   - Use Scala 3 macros to generate `person.name` instead of `productElement(0)`
   - Only if benchmarks show clear benefit

## Alternatives Considered

### A. Janino Runtime Codegen (Rejected)
- Spark's approach
- Adds dependency and complexity
- Benefits don't justify cost given existing performance

### B. ASM Bytecode Generation (Rejected)
- Lower-level than Janino
- Even more complex
- Same cost/benefit analysis

### C. Scala 3 Macro for Direct Access (Deferred)
- Could generate `person.name` instead of `productElement(0)`
- High complexity
- Save for future if benchmarks warrant
